# ContractGhost Manual Demo Artifacts

## Context
Created during founder loop on 2026-03-14 while public hosting is still blocked. The goal is to remove the last excuse in the no-link validation path by defining exactly what a screenshot-style demo should show when a freelancer replies.

## Purpose
When someone responds to outreach, send a compact visual/story demo instead of asking them to trust a vague description.

## Demo structure
Send 3 simple screenshots or mock panels in this order:

### 1. Contract upload / intake screen
Show:
- headline: "Check a freelance contract before you sign"
- upload area for PDF / DOCX
- plain-English promise: "Flags risky terms around scope, payment, IP, revisions, and termination"

### 2. Risk summary screen
Show:
- overall risk level (Low / Medium / High)
- 5 risk categories
- short bullets like:
  - "Unlimited revisions with no cap"
  - "Client owns all IP before final payment"
  - "Termination clause favors client only"
  - "Payment timing is vague"

### 3. Clause-by-clause explanation screen
Show:
- risky clause excerpt
- plain-English explanation
- why it matters to a freelancer
- suggested pushback / rewrite language

## Demo copy example
### Risk card examples
- **Scope:** Unlimited revisions with no clear limit
- **Payment:** Final payment tied to vague approval language
- **IP:** Ownership transfers before invoice is fully paid
- **Termination:** Client can exit instantly; freelancer still owes work in progress
- **Liability:** Broad indemnity clause creates open-ended exposure

### Example explanation block
**Clause:** "Contractor agrees to continue revisions until Client is satisfied."

**Why this is risky:** This can turn into endless unpaid revision rounds because there is no revision cap or acceptance rule.

**What to ask for instead:** Limit revisions to 2 rounds and define what counts as accepted work.

## Operator send format
If someone replies positively, send:
1. one-line reminder of the product
2. 1-3 screenshots/mockups
3. one question: "Would something like this have helped before you signed your last client contract?"

## Evidence goal
The demo is successful if it triggers any of:
- "Yes, this would've helped"
- a trust objection
- a feature request
- a willingness-to-pay comment
